# Gerenciador de Pacotes

> É altamente recomendado gerenciar pacotes dentro de um [[01. Ambiente Virtual (VENV)|ambiente virtual (venv)]].

## pip

Por padrão o _Python_ usa o **pip** que instala pacotes através do [Python Package Index](https://pypi.org/).  Ele é instalado junto com o _Python_.

### Instalação de pacotes

Para instalar a versão mais recente de um pacote, use o comando:

``` bash
(venv) $ python -m pip install novas

# OU

(venv) $ pip install novas
```

Você pode instalar mais de um pacote com um único comando:

``` bash
(venv) $ pip install novas requests django # instala 3 pacotes
```

Você também pode instalar uma versão específica do pacote usando o nome do pacote seguido de == e o número da versão.

``` bash
(venv) $ pip install requests==2.6.0 # instala 'requests 2.6.0'
```

### Atualização de pacotes

É possível atualizar um pacote para a versão mais recente com `pip install --upgrade`:

``` bash
(venv) $ python -m pip install --upgrade requests
# Collecting requests
# Installing collected packages: requests
#   Found existing installation: requests 2.6.0
#     Uninstalling requests-2.6.0:
#       Successfully uninstalled requests-2.6.0
# Successfully installed requests-2.7.0
```

### Remoção de pacotes

`pip unistall` seguido pelo nome de um ou mais pacotes vai remover eles do ambiente virtual.


### Outros comandos

`pip show` vai exibir informações sobre um pacote específico:

``` bash
(venv) $ python -m pip show requests
# ---
# Metadata-Version: 2.0
# Name: requests
# Version: 2.7.0
# Summary: Python HTTP for Humans.
# Home-page: http://python-requests.org
# Author: Kenneth Reitz
# Author-email: me@kennethreitz.com
# License: Apache 2.0
# Location: /Users/akuchling/envs/tutorial- env/lib/python3.4/site-packages
# Requires:
```

`pip list` vai exibir todos os pacotes instalados no ambiente virtual atual:

``` bash
(venv) $ python -m pip list
# novas (3.1.1.3)
# numpy (1.9.2)
# pip (7.0.3)
# requests (2.7.0)
# setuptools (16.0)
```

`pip freeze` vai produzir uma lista parecida com os pacotes instalados, mas a saída dele usa o formato que o `pip install` espera. Uma convenção comum é colocar essa lista num arquivo `requirements.txt`:

``` bash
(venv) $ python -m pip freeze > requirements.txt
(venv) $ cat requirements.txt
# novas==3.1.1.3
# numpy==1.9.2
# requests==2.7.0
```

O primeiro comando cria um arquivo `requirements.txt` com todos os pacotes do ambiente virtual atual especificados com a versão de cada um, possibilitando ser enviado para o repositório remoto para que outros usuários possam instalar todos os pacotes necessários com `pip install -r`:

``` bash
(venv) $ python -m pip install -r requirements.txt

# Collecting novas==3.1.1.3 (from -r requirements.txt (line 1))
#   ...
# Collecting numpy==1.9.2 (from -r requirements.txt (line 2))
#   ...
#Collecting requests==2.7.0 (from -r requirements.txt (line 3))
#   ...
# Installing collected packages: novas, numpy, requests
#   Running setup.py install for novas
# Successfully installed novas-3.1.1.3 numpy-1.9.2 requests-2.7.0
```

O **pip** têm muitas opções, você pode consultar a documentação completa dele [aqui](https://docs.python.org/3.13/installing/index.html#installing-index).

## [Pipx](https://pipx.pypa.io/stable/how-to/install-pipx/)

> Necessário Python 3.10+ instalado.

Diferente do `pip` (que instala bibliotecas no ambiente ativo), o **pipx** serve para instalar executáveis a aplicações de linha de comando _Python_ globalmente, mantendo-os isolados em seus próprios ambientes, evitando conflitos de dependência entre aplicativos executáveis como `black`, `cowsay` ou `ansible`.

### Instalação do gerenciador

``` bash
$ pip install pipx # instala o pipx

$ pipx ensurepath # configura o sistema operacional adicionando automaticamente o diretório onde o pipx armazena seus aplicativos à variável de ambiente $PATH. Isso garante que ferramentas globais (como Poetry ou HTTPie) possam ser executadas diretamente de qualquer janela do terminal.
```

### Instalação de pacotes executáveis

``` bash
$ pipx install poetry # instala a versão mais recente do poetry.

#OU

$ pipx install poetry==1.8.4 # instala o poetry 1.8.4.
```

### Atualização de pacotes

``` bash
$ pipx upgrade poetry
```

### Desinstalação de pacotes

``` bash
$ pipx uninstall poetry
```

## [UV](https://docs.astral.sh/uv/getting-started/installation/)

> É compatível com versões mais antigas de Python, mas é recomendado usar no mínimo Python 3.10.

É o substituto direto mais rápido do `pip` do mercado, sendo até 100 vezes mais rápido que o `pip`. Ele gerencia dependências, versões do _Python_ e pode atuar como um substituto para o `pipx`. Ele cria o arquivo `uv.lock` semelhante ao `.lock` criado pelo `poetry`.

### Instalação do gerenciador

Têm outras formas de instalar [aqui](https://docs.astral.sh/uv/getting-started/installation/). Mas a mais recomendada é usando `pipx` para instalar o **uv** em um ambiente isolado:

``` bash
$ pipx install uv
```

### Criando um projeto e adicionando dependências

``` bash
# 1. Cria um projeto novo  
$ uv init myapp  
$ cd myapp  
  
# 2. Initializa um novo projeto (substituto do #1)  
$ cd myapp  
$ uv init  
  
# 3. Adiciona dependências  
$ uv add requests  # OU uv pip install
  
# 4. Instala dependências (muito rápido, cached)  
$ uv sync  
  
# 5. Roda dentro do ambiente 
$ uv run main.py
```

A estrutura gerada é assim:

``` markdown
myapp  
    ├── main.py  
    ├── pyproject.toml  
    └── README.md
```

## [Poetry](https://python-poetry.org/docs/)

> Necessário Python 3.10+ instalado.

O padrão moderno para gerenciamento e empacotamento de projetos _Python_. Ele usa um arquivo `pyproject.toml` para padronizar as configurações do projeto e bloqueia as dependências com o arquivo `poetry.lock`. 

### Instalação do Poetry

Você pode instalar globalmente com:

``` bash
$ pipx install poetry
```

Também é possível instalar com `curl -sSL https://install.python-poetry.org | python3 -`, mas não é muito recomendado. O **Poetry** tende a quebrar sempre que você atualizar o _Python_, necessitando que você reinstale o **Poetry** novamente. O `pipx` cria um ambiente isolado para cada aplicação.

### Criando projeto e adicionando dependências

O **Poetry** sempre trabalha isolado da instalação global do _Python_. Para fazer isso, ele sempre vai verificar, antes de realmente executar um comando, se está rodando dentro de um [[01. Ambiente Virtual (VENV)|ambiente virtual (venv)]]. Caso esteja, irá usar ele diretamente sem criar um novo. Caso contrário, ele vai usar um dos ambientes virtuais existentes ou criar um novo para você. Você pode ver mais sobre isso [aqui](https://python-poetry.org/docs/managing-environments).

``` bash
# 1. Cria um novo projeto  
$ poetry new myapp  
$ cd myapp  
  
# 2. Inicializa um novo projeto (substituto do #1)  
$ cd my app  
$ poetry init  
  
# 3. Adiciona dependências  
$ poetry add requests  
  
# 4. Instala dependências (cria .venv automaticamente)  
$ poetry install  
  
# 5. Roda dentro do ambiente  
$ poetry run python main.py
```

A estrutura gerada é assim:

``` markdown
myapp  
    ├── myapp  
    │   └── __init__.py  
    ├── pyproject.toml  
    ├── README.md  
    └── tests  
        └── __init__.py
```

## [Conda](https://docs.conda.io/projects/conda/en/latest/user-guide/install/index.html)

Muito popular na área de _Ciência de Dados_ e _Machine Learning_. Ele não se limita a pacotes _Python_. Ele gerencia dependências de sistema (escritas em _C_ ou _C++_) e o próprio interpretador _Python_. É excelente para instalar ferramentas complexas como `TensorFlow` e garantir que compiladores em _C_ e bibliotecas estejam corretos.

O **Conda** atualmente têm 3 instaladores diferentes, você deve escolher um deles.

### [Miniconda](https://www.anaconda.com/docs/getting-started/concepts/anaconda-or-miniconda)

É um instalador mais básico, use ele caso queira instalar a maioria dos pacotes por conta própria.

### [Anaconda Distribution]([https://www.anaconda.com/docs/getting-started/concepts/anaconda-or-miniconda](https://www.anaconda.com/download))

É um instalador mais completo que vêm com uma série de pacotes voltados a _Ciência de Dados_, assim como _Navegador Anaconda_, uma aplicação GUI para trabalhar com ambientes conda.

Você pode ver as diferenças entre a Miniconda e a Anaconda Distribution [aqui](https://www.anaconda.com/docs/getting-started/concepts/anaconda-or-miniconda).

### Miniforge

É um instalador desenvolvido pela comunidade conda-forge que vêm pré-configurado para uso com o canal conda-forge. Leia mais sobre o conda-forge [aqui](https://conda-forge.org/).
